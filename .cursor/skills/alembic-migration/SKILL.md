# Alembic Migration Workflow

## Two failure modes — both are fatal

### Failure mode 1: invented revision ID

The agent writes a migration file by hand and invents a revision ID like
`a1b2c3d4e5f6`. That ID collides with an existing revision in the chain,
producing:

```
UserWarning: Revision a1b2c3d4e5f6 is present more than once
ERROR: Multiple head revisions are present for given argument 'head'
```

**Fix: never write a migration file manually. Never invent a revision ID.**
Always let Alembic generate the file to obtain a guaranteed-unique ID.

### Failure mode 2: applying the autogenerated body without reading it

The agent runs `--autogenerate` and applies the file without reading it.
In this project autogenerate is reliable for standard operations (table
creation, column addition, index changes) because Alembic owns the full
schema from scratch. However, it does not detect column type changes,
`server_default` changes, or custom constraint name changes.

**Fix: always read the generated body before applying. Confirm the
operations match your ORM change exactly. Manually add any operations
that autogenerate cannot produce. See Step 3 below.**

## Mandatory workflow

### Step 1 — Make the ORM change first

Edit the model in `irs_data/infrastructure/persistence/orm_models.py`
(or the relevant ORM file). Do not create a migration file yet.

### Step 2 — Generate the migration

```powershell
cd c:\Projects\prhrt\irs-data; .venv\Scripts\alembic.exe revision --autogenerate -m "short description of change"
```

This produces a file in `alembic/versions/` with a guaranteed-unique revision
hash. **This is the only acceptable source for a revision ID.**

The `-m` description becomes part of the filename. Keep it short and
imperative: `"add scope to ie money flow graph"`, `"add index on bill_id"`.

### Step 3 — Read the body and patch where needed

In this project autogenerate is reliable for standard operations. Read the
generated `upgrade()` and `downgrade()` functions and confirm they match your
ORM change. The body is almost always correct as generated.

Autogenerate does **not** detect these edge cases — add them manually if
your change involves them:

- Column type changes (e.g. `String` → `Text`)
- `server_default` additions or changes
- Custom constraint name changes

If the generated body contains any operation that was not part of your ORM
change, remove that operation before applying.

### Step 4 — Check the revision chain

```powershell
cd c:\Projects\prhrt\irs-data; .venv\Scripts\alembic.exe heads
```

Output must list **exactly one head** before proceeding. If two heads appear,
the new migration's `down_revision` is wrong — fix it before applying.

### Step 5 — Apply locally

```powershell
cd c:\Projects\prhrt\irs-data; .venv\Scripts\alembic.exe upgrade head
```

If this fails, fix the migration and re-run. Do not skip this step.

### Step 6 — Confirm current revision

```powershell
cd c:\Projects\prhrt\irs-data; .venv\Scripts\alembic.exe current
```

The output must show the new revision hash with `(head)`.

### Step 7 — Commit alongside the ORM change

The migration file and the ORM model change must be in the same commit.
Never commit an ORM change without its migration or a migration without
its ORM change.

## Quick reference

| Command | Purpose |
|---|---|
| `alembic revision --autogenerate -m "..."` | Generate migration from ORM diff |
| `alembic heads` | List current head(s) — must be exactly one before applying |
| `alembic upgrade head` | Apply all pending migrations |
| `alembic current` | Show which revision the DB is at |
| `alembic downgrade -1` | Roll back one migration |
| `alembic history` | Show full revision chain |

---

## What to do if the migration is wrong after applying

If the migration has been applied locally but **not** to production:

1. Roll back: `alembic downgrade -1`
2. Delete the generated file from `alembic/versions/`.
3. Fix the ORM model.
4. Re-run Step 2 to generate a fresh file with a new unique ID.

If the migration has already been applied to production:
**Do not edit or delete it.** Write a new corrective migration instead.

## Common patterns

### Add a nullable column

```python
op.add_column(
    "table_name",
    sa.Column("new_col", sa.Text(), nullable=True),
)
```

### Add a non-nullable column with server default

```python
op.add_column(
    "table_name",
    sa.Column("form_type", sa.Text(), nullable=False, server_default="990"),
)
```

### Replace a unique constraint

Before writing the `drop_constraint` call, verify the real constraint name
in the live database using the `db-query-script` skill. Never guess the name
from the ORM model — the DB name may differ.

```python
op.drop_constraint("old_constraint_name", "table_name", type_="unique")
op.create_unique_constraint(
    "new_constraint_name", "table_name", ["col1", "col2"]
)
```

### Add an index

```python
op.create_index("ix_table_col", "table_name", ["col"])
```
